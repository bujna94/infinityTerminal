# Sparkle setup

Sparkle is fully wired into the build + release pipeline. The only thing
left is a one-time manual step: copying the EdDSA **private** key from
your macOS keychain into a GitHub Actions secret so the release workflow
can sign each new DMG.

## What's already done

- SwiftPM dependency on Sparkle 2.6 (`Package.swift`).
- `Updater.swift` wraps `SPUStandardUpdaterController`.
- "Check for Updatesâ€¦" menu item wired in `main.swift`.
- `build-app.sh` embeds `Sparkle.framework` into `Contents/Frameworks/`
  and signs each helper (XPC services, Updater.app, Autoupdate) deepest-first.
- Info.plist has `SUFeedURL = https://infinityterminal.com/appcast.xml`,
  `SUPublicEDKey = zHKjehpLeorjXhF0tKWuoGYNKQUe9eYWeOQYJmQ2n5U=`
  (the public half of the EdDSA key pair, generated by Sparkle's
  `generate_keys` and stored in your login keychain),
  `SUEnableAutomaticChecks = true`, `SUScheduledCheckInterval = 86400`.
- `appcast.xml` exists at the root of `infinityTerminalWeb` (so it's
  served from `https://infinityterminal.com/appcast.xml`).
- `.github/workflows/release.yml` now has a "Sign DMG for Sparkle and
  update appcast" step that:
    1. Downloads Sparkle's CLI tools.
    2. Restores the private key from `secrets.SPARKLE_PRIVATE_ED_KEY`
       into a temp file.
    3. Runs `sign_update` to produce an EdDSA signature for the DMG.
    4. Clones the web repo, prepends a new `<item>` to `appcast.xml`,
       pushes back.
- The step **self-skips** at runtime if `SPARKLE_PRIVATE_ED_KEY` isn't
  set, so the release workflow stays green until you add the secret.

## What you have to do once

1. Open Keychain Access, search for **`ed25519`** (account name) under the
   `https://sparkle-project.org` service. That's the private key Sparkle
   generated.
2. Right-click â†’ Copy password. Or use `security find-generic-password
   -ws "https://sparkle-project.org" -a "ed25519"` in Terminal to print
   it (do NOT echo this in any other context â€” losing it compromises
   future updates).
3. Base64-encode it: `pbpaste | base64 | pbcopy`.
4. In a browser: https://github.com/bujna94/infinityTerminal/settings/secrets/actions
   â†’ New repository secret â†’ name `SPARKLE_PRIVATE_ED_KEY`, value =
   the base64'd key. Save.

After that, the next `git tag v1.x.y && git push origin v1.x.y` will:

- Build, sign, notarize, staple the DMG (existing behavior).
- Create the GitHub Release with the DMG attached (existing behavior).
- **New**: EdDSA-sign the DMG with your private key.
- **New**: Append a `<sparkle:version>` entry to `appcast.xml` in the
  web repo and push it.

Apps installed from that release onwards will see "Infinity Terminal X.Y.Z
is available" via Sparkle and self-update with one click.

## First post-Sparkle release: rollover

Existing users on v1.0.11 don't have Sparkle in their app bundle, so
they won't get an in-app update prompt for whatever the first
Sparkle-enabled release is â€” they have to install it manually one last
time. After that, all future releases auto-update.

## Backup the private key

Whatever else you do, also stash a copy of the private key somewhere
durable (1Password, Bitwarden, an offline encrypted backup). Losing it
means future versions can't deliver auto-updates that current installs
trust â€” you'd have to ship a new public key in a manually-installed
"reset" version.
